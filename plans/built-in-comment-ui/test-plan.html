Test Plan: built-in-comment-ui

Scenario: built-in-comment-ui · Plugin: plan-harness · Layer: E2E (narrative) collaboration review p0

1. Executive Summary

This document defines the end-to-end test plan for the text-selection comment layer. “Done” means a reviewer who has never seen plan-harness before can land on a shared tunnel, select text in a plan doc, leave an attributed comment, reply, and have it survive a doc regen — while the host sees unresolved counts on the dashboard, can resolve threads, and can dispatch a revise intent to an agent without leaving the browser. Coverage spans four environments (loopback, devtunnel reviewer, devtunnel host upgrade, offline), three personas (Host/Author, Reviewer, Agent), and every AC from design §7. Detailed step-level test cases live in the sibling test-cases.html; this plan carries user-journey scenarios meant for human execution or Phase-5 harness scripting.

Invariant Never trust the client

Every scenario below is written against the server-enforced contract: author is stamped server-side from the session; intent: "revise" is gated by host role; path params are allow-listed; JSONL is append-only. Any scenario that “passes” because the client withheld a field has failed.

2. Test Environment Matrix

Tests run in one of four environments. The same browser build (latest Chrome stable) serves as the reference; cross-browser sweeps are P2 and noted per-scenario where they matter.

EnvironmentPurposeAccess methodAuth path
Localhost loopback Host-side smoke runs. Exercises the loopback-bypass branch and the Host (local) sentinel author. http://127.0.0.1:<port>/view/<scenario>/<doc>.html after plan_serve_dashboard. No login form; server sees req.socket.remoteAddress as loopback and assigns role: host.
Devtunnel (reviewer) The primary production path: a reviewer on a different machine, authenticated via the per-share password, writing comments. Tunnel URL from plan_share; cookie set after login form. Password + display name form; server stamps req.user.name, role: "reviewer".
Devtunnel (host session upgrade) Rare but real: the host is mobile, authenticates through the tunnel, and must see the two-button composer (Reply + Request update). Same tunnel URL, password login, plus a server-side role flag (e.g. plan_share --host-name alice). Login + flagged display name; server assigns role: "host".
Offline (laptop lid closed) Draft durability under network flap. Simulates tunnel dropout by closing the laptop mid-compose or throttling DevTools to Offline. Devtunnel URL, then simulate disconnect; draft restore on reload. Existing session cookie; reauth on expiry.

Fixture data: a fresh scenario test-cmt-e2e with one pre-generated design.html containing exactly three [data-section-id] ancestors (§1, §2, §3) and one code block. Seeded .comments/design.jsonl carries five lines for pagination / regen checks; empty-state cases use a new doc with zero comments.

3. Personas

Host / Author — “Ava”

Mental model: runs plan-harness from VS Code; owns the scenario; treats comments as the inbox for the doc she wrote. Confident with keyboard shortcuts and expects reviews to reduce to actionable diffs.

Capability tier: technical; knows the filesystem layout under plans/<scenario>/; will grep JSONL if she suspects a bug.

Likely errors: triggers Ctrl+Shift+Enter by muscle memory intending Reply; expects Request update to always fire synchronously and gets confused by passive mode on first run. Assumes loopback bypass works while tunneled (it does not).

Reviewer — “Raj”

Mental model: opens a link a teammate pasted in chat, logs in with a display name and a password dropped in the message. Expects the feel of Notion / Google Docs / Hypothesis: select, comment, move on.

Capability tier: non-technical from the plugin’s perspective — may be a designer, PM, or SME. Does not know what JSONL is. Will close the tab if the first comment takes longer than 30 seconds.

Likely errors: forgets to select before clicking (no button appears, gives up); types in the wrong browser tab after context-switching; loses a draft by hitting Esc and expects auto-save.

Agent persona — “Writer subagent” future-ready

Mental model: invoked by /plan-revise or by the active-mode dispatcher. Receives { anchor, exact, body } context and writes a unified diff into .comments/<doc>.proposals/<cmtId>.diff.

Capability tier: deterministic per prompt; no UI interaction. Failures show as a stuck Dispatched chip rather than a crashed page.

Likely errors: proposes a diff whose target span drifted since dispatch (anchor mismatch on apply); writes a diff that would alter an unrelated section; runs beyond the 3-concurrent-per-scenario cap.

4. Golden-Path Scenarios P0

Six narrative scenarios that, passed together, mean the core value prop ships. Each opens by default so a dry-run can be done top-to-bottom.

E2E-1 Reviewer leaves the first comment on an empty doc P0devtunnel

Goal

Confirm TTFC p50 < 30s, empty-state discoverability, and attribution integrity for a brand-new reviewer.

Preconditions

  • Fresh scenario test-cmt-e2e, .comments/design.jsonl does not exist.
  • plan_share active; reviewer has URL + password in a chat message.
  • Host is NOT watching the page (isolates reviewer discoverability).

Steps

  1. Reviewer clicks tunnel URL; login form renders; types display name Raj + password.
  2. Login redirects to the target plan doc. First-visit tooltip appears near the doc body: “Select any text to comment”.
  3. Reviewer drags across a sentence inside §2; within 150 ms a floating Comment button renders at the selection end.
  4. Reviewer clicks the button; right-side panel slides in (~380 px), composer takes focus, the selected span is highlighted with <mark>.
  5. Reviewer types “This conflates two concepts” and hits Ctrl+Enter.
  6. Thread card appears in the sidebar with author Raj, relative timestamp, and the unresolved indicator; empty-state card disappears.

Expected outcome

Sticky doc pill now reads 1 unresolved. The sidebar TODO panel remains empty (this is a comment, not a TODO). The reviewer experiences at most one mouse gesture + one keystroke sequence between login and posted comment.

Data left behind on disk

One line in plans/test-cmt-e2e/.comments/design.jsonl with server-stamped id, createdAt, author: "Raj", anchor.sectionId matching the [data-section-id] of §2, non-empty prefix/exact/suffix, intent: "comment", resolved: false. No proposal directory is created.

E2E-2 Host resolves a thread on loopback P0localhost

Goal

Confirm resolve semantics (AC-4 interim: host-only), undo window, append-only event append, and SSE broadcast.

Preconditions

  • Scenario has one unresolved comment by Raj.
  • Host opens the same doc on http://127.0.0.1:<port>; /api/me returns { role: "host", name: "Host (local)" }.
  • A second browser tab on the tunnel URL is open as Raj, subscribed to the SSE stream.

Steps

  1. Host opens the sidebar and clicks Resolve on Raj’s thread.
  2. A toast with a 5-second undo countdown appears; the thread visually dims but is not yet collapsed.
  3. Host waits the full 5 seconds without clicking Undo.
  4. The thread collapses; a Resolved badge replaces the unresolved dot.
  5. In Raj’s tunnel tab, within 1 second, the SSE event flips the thread to resolved without a page reload.

Expected outcome

The doc pill unresolved count decrements by 1. The dashboard scenario card unresolved count decrements on next load. Host audit log on stderr emits [comments] PATCH <id> field=resolved value=true actor=Host (local).

Data left behind on disk

A second JSONL line appended with the same id, resolved: true, server-stamped resolvedBy and resolvedAt. The original comment line is untouched — last event wins on read.

E2E-3 Reviewer drafts a reply, closes laptop, returns next day P0offline

Goal

Confirm UC-12: draft persistence under tunnel dropout and composer restoration on reload.

Preconditions

  • Reviewer is logged in with a live session cookie.
  • DevTools Network panel available to toggle Offline, or a physical laptop lid closure is acceptable.

Steps

  1. Reviewer selects a span in §3, opens the composer, types a 300-char reply.
  2. Reviewer toggles DevTools Network to Offline (simulates tunnel drop). Banner appears: “You’re offline. Drafts saved locally.” with role="status" aria-live="polite".
  3. Reviewer hits Ctrl+Enter; POST fails; client retries on backoff 1s / 2s / 4s, each failure surfaces in the banner; draft text remains visible.
  4. Reviewer closes the tab entirely.
  5. 24 hours later, reviewer reopens the URL (session cookie still valid), navigates to the same doc.
  6. On load, the client scans localStorage keyed by (scenario, doc, anchor), finds the draft, re-highlights the original selection, and re-opens the composer pre-filled with the 300 chars.
  7. Reviewer hits Ctrl+Enter; POST succeeds; draft cleared from localStorage.

Expected outcome

Zero keystrokes lost. Zero silent drops. The re-highlight lands on the exact original span (anchor walker re-runs because the doc may have regenerated).

Data left behind on disk

One new JSONL line on success. localStorage key removed. No proposals, no tombstones.

E2E-4 Doc regenerates, anchors hold under ≤20% section churn P0localhost

Goal

Confirm AC-3: ≥ 90% re-anchor rate (exact + fuzzy combined) when section IDs are preserved and per-section text churn is under 20%.

Preconditions

  • Scenario has 20 comments seeded across three sections with varying exact lengths (8–120 chars).
  • Host has a modified source that, when rendered, produces the same sectionIds but edits one sentence per section (< 20% delta).

Steps

  1. Host runs plan_reanchor test-cmt-e2e design.html.
  2. Reanchor cascade runs per comment: exact prefix+exact+suffix within the same sectionId, then section-scoped exact, then Jaccard ≥ 0.72.
  3. Host reloads the doc.
  4. Each comment card renders a status chip: anchored, fuzzy, or orphaned.

Expected outcome

At least 18 of 20 comments (≥ 90%) land as anchored or fuzzy. Orphans — at most 2 — appear in the pinned Needs reattachment group at the top of the sidebar with the original exact shown in a <blockquote>. stderr: [comments] reanchor test-cmt-e2e/design held=18 migrated=2 orphaned=2 (or similar within ≥ 90% target).

Data left behind on disk

One reanchor event appended per moved comment (migratedFrom captures the previous anchor). No orphan event is written; orphan status is derived on read.

E2E-5 Author requests revise (passive mode) then dispatches via /plan-revise P0localhost

Goal

Confirm UC-13 and UC-15: end-to-end revise loop with the default reviseMode: "passive", covering the author-only button gate, JSONL state transitions, and accept-apply safety.

Preconditions

  • .comment-config.json absent (default passive).
  • Host on loopback; /api/me returns role: "host"; composer submit row shows Reply and Request update buttons.
  • A reviewer account is NOT open concurrently (isolates the host path).

Steps

  1. Host selects a sentence in §3 of design.html.
  2. Host types “Data model section conflates anchor and section id; please split” and clicks Request update.
  3. Card renders with a Pending dispatch chip and a Dispatch now button.
  4. Host exits the browser, runs /plan-revise test-cmt-e2e from the CLI.
  5. The skill lists pending comments, calls revise-dispatch per comment; chips flip to Dispatched; each comment hands off to the writer subagent.
  6. Writer subagent writes .comments/design.proposals/<cmtId>.diff; server appends a proposed event; SSE pushes the update.
  7. Host reopens the browser; the card shows a Proposal ready chip with a notification toast.
  8. Host clicks View diff; modal overlay renders unified diff with a live preview; host clicks Accept.
  9. Server applies the diff to the source, appends an accepted event, flips resolved: true, broadcasts SSE.

Expected outcome

The section text visibly changes on next doc load. Card shows Accepted chip + resolved badge. Exactly one durable mutation to the doc source per accepted proposal. Reviewer tabs never see the Request-update button, the proposal chip details, or the diff modal.

Data left behind on disk

JSONL trail: pending → dispatched → proposed → accepted + resolved: true events for the same id. One .diff file preserved under .comments/design.proposals/. Doc source updated at the precise anchor target.

E2E-6 Host reads dashboard badges, drills into unresolved thread P0localhost

Goal

Confirm UC-3 / UC-8: dashboard-to-doc discoverability. The host should never have to hunt for an unresolved thread.

Preconditions

  • Scenario has three docs: one with zero unresolved, one with two unresolved (including one orphan), one with five unresolved.
  • Host’s localStorage read cursor is before the most recent comment.

Steps

  1. Host opens the dashboard.
  2. Scenario card shows +7 new / 7 unresolved (orphans excluded from the badge per design §6.9).
  3. Host clicks the scenario; per-doc breakdown lists the three docs with respective counts.
  4. Host clicks the doc with 5 unresolved; sidebar auto-opens with threads expanded; first unread is pinned to viewport top with a 1.2-second accent flash.
  5. Host reads and does not resolve; closes the tab.

Expected outcome

On next dashboard visit, the card shows 5 unresolved (no “new” overlay, because the host’s read cursor advanced). Orphan count is never part of the badge. Sticky doc pill is keyboard-focusable.

Data left behind on disk

No JSONL writes. Cursor update is localStorage-only (host identity). TODO: confirm cursor key format stabilizes before shipping so multi-device hosts don’t get “everything new” on a second machine.

5. Edge & Failure Scenarios P1

Eight scenarios that cover the hazards enumerated in design §10. Start collapsed; expand when running a targeted suite.

E2E-7 Reviewer hits the rate limit P1devtunnel

Goal

Confirm AC-7 and R-2 mitigation: visible countdown, disabled submit, accurate error code on the wire.

Preconditions

  • Server rate limit set to 10 posts / 60s / session.
  • Reviewer has posted 10 comments in the last 55 seconds.

Steps

  1. Reviewer selects text and types an 11th comment.
  2. Reviewer hits Ctrl+Enter.
  3. Server returns HTTP 429 with a Retry-After header equal to the remaining window.
  4. UI shows an inline error: “Slow down — try again in 7s” with a live countdown.
  5. Submit button is disabled until the countdown reaches zero; textarea content is preserved.
  6. Reviewer waits; submit re-enables at zero; Ctrl+Enter now succeeds.

Expected outcome

No JSONL write during the throttled window. stderr audit emits one [comments] rate-limit sid=... remaining=0s line. No client-side reload required.

E2E-8 Path-traversal attempt is rejected at the route layer P1security

Goal

Confirm AC-5 and R-3: no read/write outside plans/<scenario>/.comments/, and that the server logs the rejection.

Preconditions

  • An authenticated session cookie in hand; curl or similar.

Steps

  1. Attacker issues POST /api/comments/..%2F..%2Fetc%2Fpasswd/design with a valid body and cookie.
  2. Server response: HTTP 400 with JSON body { "error": "invalid_scenario", "message": "Scenario name must match ^[a-zA-Z0-9_-]+$" }.
  3. Repeat with a backslash variant: POST /api/comments/test/..%5C..%5Csecret → same 400.
  4. Repeat with a URL-encoded null byte: POST /api/comments/test%00/design → same 400.
  5. Repeat with an absolute Windows path fragment: POST /api/comments/test/C%3A%5Cplan → same 400.

Expected outcome

No filesystem reads or writes outside the scenario’s .comments/ directory for any of the four vectors. stderr audit emits [comments] BLOCKED path-traversal param=<raw> ip=<...> per attempt. 400 error message never echoes the unescaped input back into the response body.

E2E-9 Concurrent POSTs from 50 reviewers do not interleave P1devtunnel

Goal

Confirm AC-10 and R-4 mitigation: durable append under concurrency, no partial lines, no lost events.

Preconditions

  • 50 authenticated sessions (script-driven); each holds a distinct author.
  • Target doc is empty of comments.

Steps

  1. Script fires 50 parallel POST /api/comments/<scenario>/<doc> requests within a 200-ms window.
  2. All 50 responses captured; each expected to be HTTP 201.
  3. Server writes happen under O_APPEND with a per-file mutex and fsync.
  4. Script reads back design.jsonl and parses line-by-line with JSON.parse.

Expected outcome

Exactly 50 valid JSON lines. Zero parse errors (no torn writes). 50 distinct id values. 50 distinct author values as expected. createdAt monotonic non-decreasing.

E2E-10 Disk-full yields 507, not silent loss P1critical

Goal

Confirm AC-15 and R-4: ENOSPC translates to an HTTP 507 Insufficient Storage plus a clear UI message. A 200 OK without a durable write is a bug, not an optimization.

Preconditions

  • Test ramdisk or tmpfs mount capped at 1 MB, holding .comments/.
  • Seed design.jsonl with 999 KB of valid lines.

Steps

  1. Reviewer posts a normal comment.
  2. Server attempts appendFile; the underlying write fails with ENOSPC.
  3. Server does NOT retain a partial write; any buffered bytes are discarded.
  4. Server responds HTTP 507 with JSON body { "error": "disk_full", "message": "Host is out of disk; comment not saved." }.
  5. Client renders an inline error: “Couldn’t save — disk full on host”; composer content preserved; submit re-enabled after 2s.

Expected outcome

Zero new JSONL lines. stderr audit: [comments] ENOSPC <scenario>/<doc> author=.... No partial line at end of file. Reading the file with plan_get_files after the failure shows exactly the pre-test state.

E2E-11 SSE stream reconnects during a 15-second network flap P1offline

Goal

Confirm the SSE registry survives a transient disconnect; no duplicate events; no missed events after reconnection.

Preconditions

  • Reviewer tab A subscribed to SSE on /api/comments/<scenario>/<doc>/events.
  • Reviewer tab B authenticated and ready to post.

Steps

  1. In tab A, DevTools Network panel set to Offline for 15 seconds.
  2. During the flap, tab B posts two comments; both appear in JSONL with server-assigned ids.
  3. Tab A restored to online.
  4. EventSource auto-reconnects; client passes Last-Event-Id header.
  5. Server replays the two missed events.

Expected outcome

Tab A’s sidebar shows exactly two new thread cards with correct ids and authors. No duplicate cards (ids dedup on render). Sticky pill count increases by 2. stderr: one SSE client disconnected and one SSE client connected pair; no crash.

TODO: confirm Last-Event-Id plumbing in comment-manager.js before this scenario can be run as-is — design §4 mentions SSE but does not pin the resume protocol.

E2E-12 Reviewer attempts revise; server refuses with 403 P1security

Goal

Confirm the permission matrix in design §9: a reviewer cannot forge intent: "revise".

Preconditions

  • Reviewer session, role: "reviewer".
  • Reviewer has a valid anchor for a span in §2.

Steps

  1. Reviewer opens DevTools and issues fetch('/api/comments/<scenario>/<doc>', { method: 'POST', credentials: 'include', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ anchor, body: 'forge me', intent: 'revise' }) }).
  2. Server checks role BEFORE appending; reads the role from the session, not from the body.
  3. Response: HTTP 403 with JSON { "error": "role_required", "message": "Only the host may request an update." }.

Expected outcome

Zero JSONL writes. No proposal directory created. stderr audit: [comments] BLOCKED intent=revise actor=<reviewer>. The reviewer UI never rendered the button, confirming defense-in-depth: the client had no affordance AND the server re-checks.

E2E-13 Edit window expires mid-compose P1devtunnel

Goal

Confirm AC-11: the 10-minute author edit window is enforced server-side, and the UI announces expiry gracefully.

Preconditions

  • Reviewer posted a comment at t0.
  • Clock advanced to t0 + 9m 55s; reviewer clicks Edit.

Steps

  1. Edit inline editor opens with the original body; focus on the textarea.
  2. Reviewer types for 20 seconds (clock is now t0 + 10m 15s).
  3. Client sends PATCH /api/comments/<scenario>/<doc>/<id>.
  4. Server compares now - createdAt against the 10-minute budget; returns HTTP 403 { "error": "edit_window_expired" }.
  5. UI surfaces an inline message: “Edit window closed. Reply to the thread instead.” with a Copy draft button so the user doesn’t lose the text.

Expected outcome

Original JSONL line unchanged. No new edit event appended. Draft text is recoverable via Copy draft; no surprise data loss even on a server-side reject.

E2E-14 Orphan reattach flow after aggressive regen P1localhost

Goal

Confirm UC-7 / AC-9: a reviewer (any authenticated user) can reattach an orphan by selecting the new target.

Preconditions

  • One comment in test-cmt-e2e whose anchor did not survive regen (forced by editing the relevant sentence beyond fuzzy threshold).
  • Pinned Needs reattachment (1) group visible at the top of the sidebar, amber dot.

Steps

  1. User clicks the orphan card; original anchor.exact renders in a <blockquote>; sidebar enters Reattach to selection mode with an instruction banner.
  2. User selects a new span in the regenerated doc that matches the original intent.
  3. User clicks Confirm reattach.
  4. Client POSTs a reanchor event with the new prefix/exact/suffix and new sectionId.
  5. Server appends the event; orphan group empties; thread card moves back into the sidebar body.

Expected outcome

Orphan count drops to 0. A new JSONL event carries the updated anchor; the original anchor data is preserved in earlier events (append-only). Alt path: clicking Archive appends an archived event instead; thread hides without a new anchor, and clicking “Show archived” renders it back.

6. Cross-cutting Concerns

Accessibility

Checks from UC-10 and AC-8. Run with the browser’s OS-native screen reader (NVDA on Windows, VoiceOver on macOS). Target WCAG 2.1 AA.

  • Keyboard-only create: tab to a section, enter reading mode, select by word with Shift+arrow, press Ctrl+Alt+M; composer gains focus; submit with Ctrl+Enter; focus returns to selection end on close.
  • SR announcements: on composer open, SR reads “Comment composer, referencing: <selected text>”. On successful post, an aria-live="polite" region reads “Comment posted by <author>”.
  • Focus choreography: composer open steals focus; sidebar open/close does not. Modal proposal dialog (host) traps focus and returns it to the opening chip on close.
  • Contrast: highlight color ≥ 3:1 against underlying text background in both themes; sidebar thread card body text ≥ 4.5:1 against var(--surface).

Security

Every scenario in §5 that is tagged security plus these standing checks:

  • AC-5 sweep: run the published path-traversal attack list (min: .., ../, ..%2F, ..%5C, absolute paths, null bytes, long names > 80 chars) against all five comment routes and the three revise routes. All must 400 or 403.
  • AC-6 XSS sweep: post 12-payload list including <img src=x onerror=alert(1)>, <svg onload>, javascript: URIs, broken tag combos, markdown-looking HTML, and CJK full-width variants. All render as escaped text; no alert dialog fires. The same escape applies to author values.
  • CSRF posture: open a cross-origin page; attempt fetch('/api/comments/...', { method: 'POST', credentials: 'include' }). Browser blocks via SameSite=Strict; no request ever reaches the server. Record the behavior as a baseline for future session flag changes.
  • Session TTL: force-expire the cookie (rolling TTL from auth.js); next POST returns 401; client redirects to login without data loss (draft retained).

Performance

Targets drawn from design §1 Success Metrics.

  • TTFC: instrument login-submit → first-comment-201. p50 < 30s, p95 < 90s across a 10-reviewer synthetic cohort.
  • Anchor scan cost: open a doc with 100 comments; measure SelectionDetector initialization and first-paint-to-all-highlights duration. Target < 200 ms on reference hardware; main thread blocked < 50 ms any single frame.
  • Reanchor batch: on a 100-comment doc regenerate, plan_reanchor completes in < 500 ms end-to-end. Guards against accidental O(N²) matchers.
  • SSE idle cost: a connected reviewer sitting for 10 minutes incurs no more than one keepalive round-trip per 30 seconds and no measurable memory growth.

9. Not in Scope

Deliberate omissions. Revisit when the linked design item promotes from P2 to P0/P1, or when a success metric starts missing.

  • PDF export with margin notes (UC-9, AC-13). P2 in design §7; will get its own test sub-plan when the pipeline lands.
  • Multi-language comment content. The MVP stores UTF-8 raw and escapes on render; RTL rendering, language-aware anchoring, and translation are out of scope.
  • Scaling beyond 100 comments per doc. Design §10 R-10 acknowledges JSONL grows unbounded; 100 comments is the tested upper bound. 1k / 10k / 100k sweeps are reserved for a follow-on perf scenario.
  • Cross-browser matrix beyond Chrome stable. Firefox, Safari, Edge-specific quirks (Selection API edge cases, SSE reconnect timers) are deferred to a P2 sweep.
  • Agent personas authoring real comments. Schema supports it; MVP writes are human-only. When an agent persona starts writing, it needs its own trust/attribution scenario.
  • Active-mode concurrency burst > 3 per scenario. The queue-full banner is specified in UC-14 but its test is reserved for the Phase-5 harness since it requires synthetic agent jobs.
  • PATCH-body edit history UI. Edits are append-only on disk (AC-11 tested), but a diff-style history viewer is P2 and not part of this plan.
