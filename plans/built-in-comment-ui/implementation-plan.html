1. Status Snapshot

On track 3 of 11 phases shipped; 8 open

Stable section IDs, the server-side comment CRUD / SSE / rate-limit stack, and the read-only sidebar panels (TODOs + Comments) are all in main. What remains is the browser write-path: selection detection, composer, thread rendering, orphan recovery, revise intent with its agent dispatcher and proposal modal, the re-anchor cascade MCP tool, and the hardening sweep.

Current blocker: the user. Two open decisions from design §2 gate coding — who can resolve a thread (any-auth vs host-only) and the fuzzy re-anchor algorithm (Jaccard 0.72 vs Levenshtein 0.85). Phase 6 (threads) depends on the first; Phase 10 (re-anchor) depends on the second. No line of production code should ship until both are decided.

Open decision Fuzzy algorithm pick

Architect: Jaccard token-overlap ≥ 0.72. PM: Levenshtein similarity ≥ 0.85. Pick before writing tier3Match. Jaccard is O(n+m) and cheap; Levenshtein is O(n·m) and catches small edits better. A 250-sentence stress sample should settle it; if both come within 3 pp on survival rate, pick Jaccard for simplicity.

Pseudo-code — reanchorDocument(scenario, doc)

const comments = await listComments(scenario, doc);
const html      = await readFile(docPath);
const sections  = parseSections(html); // { [sectionId]: text }

for (const c of comments) {
  if (!c.anchor || c.anchor.orphaned) continue;
  const { sectionId, exact, prefix, suffix } = c.anchor;
  const sect = sections[sectionId];

  // tier (a) -- exact prefix+exact+suffix in same section
  if (sect && sect.includes(prefix + exact + suffix)) { heldCount++; continue; }

  // tier (b) -- bare exact in same section
  if (sect && sect.includes(exact)) {
    appendEvent({ id: c.id, anchor: { ...c.anchor, migratedFrom: sectionId } });
    migratedCount++; continue;
  }

  // tier (c) -- fuzzy across all sections
  const best = bestFuzzyMatch(exact, sections); // returns { sectionId, score }
  if (best.score >= THRESHOLD) {
    appendEvent({ id: c.id, anchor: { ...c.anchor, sectionId: best.sectionId, migratedFrom: sectionId } });
    migratedCount++; continue;
  }

  // tier (d) -- orphan
  appendEvent({ id: c.id, anchor: { ...c.anchor, orphaned: true } });
  orphanCount++;
}

console.error(`[comments] reanchor ${scenario}/${doc} held=${heldCount} migrated=${migratedCount} orphaned=${orphanCount}`);
return { held: heldCount, migrated: migratedCount, orphaned: orphanCount };

7. Follow-up Risks

Risks the design doc under-weighted, surfaced while sequencing the phases:

R-FU-1 — injectSidebarPanels vs SelectionDetector DOM order

Phase 3 inserts sidebar DOM on DOMContentLoaded. Phase 4's selection detector walks ancestors for [data-section-id]. If the sidebar injection re-parents main content or wraps it in a shadow DOM, the ancestor walk breaks. Mitigation: the selection detector must bind on document, not on a cached .container reference, and it must skip any selection whose commonAncestorContainer is inside nav.side-nav.

R-FU-2 — window.__PLAN_HARNESS_META__ cache staleness

Phase 1 emits the meta object at render time. If the host changes their display name mid-session, the widget keeps the old name until reload. This is an implicit dependency of AC-2; the server stamps the truth, so no incorrect data is persisted, but the composer may show the wrong reviewer name in optimistic renders. Mitigation: always read author off the POST response, never off __PLAN_HARNESS_META__, when rendering the new thread card.

R-FU-3 — proposal modal shadows existing theme toggle

The theme toggle is position: fixed; z-index: 10000. The Phase 9 modal is also a dialog at high z-index. If both paint at once, clicking the toggle through a modal scrim could blur focus out of the modal and break the focus-trap invariant. Mitigation: modal scrim uses z-index: 10500, theme toggle stays at 10000, scrim blocks pointer events on the toggle.

R-FU-4 — /plan-revise batching on large queues

Design assumes a handful of pending revises. A long review cycle could accumulate 30+. The current spec dispatches them in sequence without inter-task delay, which bursts the subagent runtime and risks rate-limit exhaustion at the agent-model layer. Mitigation: add a configurable batch size (default 5) in .comment-config.json; dispatcher yields between batches.

R-FU-5 — .comments/ in plugin's own repo

Per the self-dev loop (Phase 11), running the plugin against its own plans dir writes JSONL under this very repo's plans/built-in-comment-ui/.comments/. These become part of the plugin's history unless excluded. Mitigation: append plans/**/.comments/ and plans/**/*.proposals/ to the repo's .gitignore during Phase 11.

R-FU-6 — proposal apply races regen

Phase 9's Accept path re-runs the cascade before applying the diff. If a regen happens between proposal-written and accept-clicked, the anchor may have migrated twice and the diff's hunk headers no longer match. Mitigation: the accept endpoint should match by anchor.exact rather than by line numbers; if no unique match exists, refuse with 409 and surface the re-anchor state in the modal so the author sees why.
