State Machine: built-in-comment-ui

Scenario: built-in-comment-ui · Plugin: plan-harness · collaboration review event-sourced

1. Overview

The comment system is event-sourced: every user-visible mutation is an append-only JSONL line (see Design §3 Storage Layout). "State" therefore is always a fold over an ordered event stream — the last event of a given kind wins on read. This document enumerates every entity in the design whose fold has more than one possible value, pins down the transitions that produce those values, and lists the invariants that must hold at any fold.

Six state machines are in scope:

#EntitySource of truthWhy the state matters

2Comment.intent="revise"JSONL event log (reviseStatus events)Separate lifecycle for agent-assisted edits; gates when a diff is writable and when it can be applied.
3AnchorRe-anchor cascade outputDetermines whether the highlight still binds to real text or floats in the orphan bucket.
4Session (auth.js)In-memory Map + HMAC cookieEvery mutation above checks the session first. Expired or revoked sessions must cleanly reject, not leak.
5SSE clientIn-memory Map<docKey, Set<Response>>A disconnected client must not silently miss events; auto-reconnect behaviour is part of the contract.
6Rate-limit bucketIn-memory Map<sid, { tokens, lastRefill }>Token bucket protects disk and keeps the devtunnel honest; states drive 429 vs 201 responses.

2. Conventions & Legend

Every diagram below uses the same visual grammar. Colour follows the Linear palette from Design §6 Theme Parity: var(--accent) for normal transitions, var(--green) for terminal/successful outcomes, var(--yellow) for degraded but recoverable, var(--red) for error / terminal failure, var(--muted) for auto/timer-driven transitions. Every state label is reachable from the start node; the lone exception is a labelled must-not-happen dotted edge, which is drawn explicitly to prove the tester has a negative case for it.

normal transition success / terminal ok degraded / retryable error / terminal fail auto / timer

Transition table format: From | Event | Guard | To | Side effects. The Event column names an API route, SSE payload, or timer; the Guard column names the condition that must hold for the transition to fire (e.g. actor.id === comment.author); Side effects lists every durable or observable consequence (JSONL line appended, SSE broadcast fired, stderr audit line, etc.).

Empty / no-op transitions: a table row with a red "guard fails" badge means the event is received but the guard rejects it. The only permitted observable effect in that row is an HTTP error response — never a log line, never a JSONL append, never a state change. This is captured once here and referenced (not repeated) per-entity.

3. Comment Lifecycle

Entity
Comment

A comment starts life as a pure client-side draft, becomes an id-bearing JSONL line once the server accepts the POST, then accrues further events (edit, resolve, unresolve, delete). Because every event is append-only, "state" here is the fold of the latest relevant event. Orphan status is an orthogonal flag set by the re-anchor cascade (see §5) and can toggle on any non-deleted comment.

3.1 States & Diagram

Primary states: draft (client only, never on disk), posted (JSONL line present, resolved=false, deleted=false), edited (latest event has a new body from the author within the edit window), resolved (latest fold has resolved=true), unresolved (resolved then un-resolved — a fresh resolved=false event appears after a resolved=true), deleted (soft-delete marker present — terminal for the primary axis). The orphan flag is drawn as a dotted orthogonal lane.

Figure 3.1. Comment lifecycle. deleted is terminal. orphan is an orthogonal flag: it toggles on any non-deleted state but never replaces the main state.

3.3 Invariants

Invariants Comment FSM
  • I-C1. Delete is terminal. Once a deleted=true event exists for a comment id, no later event may move that id to any non-deleted state. The server rejects any mutation on a deleted id with 404.
  • I-C2. Append-only log. Resolve and unresolve are two separate events — never an in-place toggle. Reading history always yields a total ordering.
  • I-C3. Author attribution is server-controlled. author is always taken from req.user.name (or "Host (local)" on loopback); client-supplied author in the request body is silently dropped. See Design §9 Attribution Integrity.
  • I-C4. Body is raw, escape on render. The body string is stored verbatim (1-4000 chars); every renderer uses textContent or equivalent HTML escape. No state transition changes this contract.
  • I-C5. Orphan is orthogonal. orphan=true can coexist with any of posted/edited/resolved/unresolved. It can never coexist with deleted (the server rejects re-anchor events targeting a deleted id).
  • I-C6. Drafts never leak attribution. A draft lives only in the reviewer's localStorage; no author, createdAt, or id exists on it. If the browser crashes, re-opening the doc re-hydrates the draft without ever crossing a trust boundary.
  • I-C7. Reply does not mutate parent. Creating a reply is a new posted comment with replyTo set; the parent's fold is unchanged. Resolving a parent does not resolve its replies (they inherit visibility via UI collapse, not state).

3.4 Open Questions / Edge Cases

  • [ ] AC-4 resolve permission. PM says host-only; Architect says any auth user. Until this is pinned, the transition table row for PATCH resolved is dual-labelled. Design AC-4.
  • [ ] Edit-window length (30s? 5min? unbounded for author?). Design §3 says "within window" but does not pin a number. Implication: the guard on the edited transition is under-specified.
  • [ ] Should editing a resolved comment's body auto-unresolve it, keep resolved, or be forbidden? Current table assumes PATCH body is allowed on resolved and keeps the flag; needs confirmation.
  • [ ] What should happen to replies when the parent is deleted? MVP: replies stay; deleted body shows "[deleted]". An alternative is cascading delete. TODO: pick one and add a test.
  • [ ] Host-driven hard-delete (purge) — not MVP per §10 R-10, but its entry point is worth reserving.

4. Revise-Intent Lifecycle

Entity
Comment.intent === "revise"

When an author clicks Request update (Ctrl+Shift+Enter), a comment is created with intent: "revise" and a separate status axis, reviseStatus. This lifecycle runs in parallel to the Comment FSM: the underlying comment can still be edited or deleted; the revise axis independently tracks whether the author is awaiting a proposal, has one to review, or has accepted / rejected it. Passive mode (default) requires a manual dispatch; active mode dispatches synchronously on POST.

4.1 States & Diagram

States: pending (JSONL line written; no dispatch yet), dispatched (subagent enqueued or running), proposed (diff file written at .comments/<doc>.proposals/<cmtId>.diff; reviseProposalRef set), accepted (diff applied to doc; parent comment resolved=true), rejected (no doc change; parent comment retained for reply). An additional guard — anchor-drift check — fires on accept: if the anchored span no longer matches, the transition refuses and the comment falls into orphan (on its Comment FSM axis) with the proposal preserved.

Figure 4.1. Revise lifecycle. Active mode fast-paths through pending; passive mode requires an explicit dispatch. Both terminate at accepted or rejected; an anchor-drift check at accept time protects the doc from being patched against stale text.

4.3 Invariants

Invariants Revise FSM
  • I-R1. Host gate at every entry. intent:"revise" can only land if the server-side role check passes. A client that sets intent:"revise" as a non-host actor is rejected 403; no JSONL event, no state change. See Design §9 Permission Matrix.
  • I-R2. accepted and rejected are terminal. There is no route that moves an accepted comment back to proposed, and no route that re-dispatches a rejected one. A new follow-up revise is a new comment.
  • I-R3. Anchor-drift guard is always checked at accept. The server re-runs the prefix+exact+suffix match against the current doc before applying the diff. This guard fails open into orphan on the Comment FSM, never silently patches stale text.
  • I-R4. One doc mutation per proposal, at most. Accepting a proposal applies its diff exactly once; a second accept is 404.
  • I-R5. reviseProposalRef only valid from proposed onward. Before proposed, the field is null. The server refuses to serve the diff file unless the fold has reviseStatus === "proposed" | "accepted" | "rejected" and actor is host.
  • I-R6. Active mode respects the concurrency cap. At most 3 concurrent dispatched per scenario; the 4th request falls to pending with a UI banner (UC-14). This cap is enforced in revise-dispatcher.js, not in the route.
  • I-R7. Revise and Comment FSMs compose, not interfere. A revise comment can be deleted (Comment FSM terminal) while still in proposed; the proposal file is kept for audit but can never re-enter the accept path because the parent is gone.

4.4 Open Questions / Edge Cases

  • [ ] Agent-failure transition from dispatched: does it fall back to pending, to a new failed terminal, or surface as a PATCH-body edit on the underlying comment? MVP table assumes fall-back to pending; needs review.
  • [ ] What happens if the author edits the comment body while it is dispatched? Does the agent see the new body (re-dispatch implicit) or keep working on the snapshot it received?
  • [ ] Concurrency cap semantics for passive mode: no cap today — a /plan-revise run could enqueue 100 jobs at once. Decision needed whether to apply the 3-per-scenario cap there too.
  • [ ] Should a reviewer be able to see the proposal diff? Design says author-only; no route exposes the file to remote sessions. Confirm in test cases (negative test: remote GET on proposal file returns 403).
  • [ ] Can revise-reject append a reason string? Current design has no such field; add to Comment as rejectReason if useful, or land as a follow-up reply.

5. Anchor Re-Anchor Cascade

Entity
Anchor (after regen)

When a plan doc is regenerated, every existing comment's anchor is tested against the new HTML by a three-tier cascade (Design §4 Re-anchor sequence). Each anchor ends in exactly one of three states: anchored (tier a held), migratedFrom (tier b or c matched; the anchor is now attached to a different section or span), or orphaned (tier d — no acceptable match). The cascade is deterministic: tier order is never shuffled, and each anchor is touched exactly once per regen.

5.1 States & Diagram

Figure 5.1. Anchor cascade. Each anchor traverses tiers (a) then (b) then (c); the first hit wins. Only (d) — no hit — produces orphaned. Tier (b) preserves sectionId; tier (c) may change sectionId and therefore records migratedFrom.

5.3 Invariants

Invariants Anchor FSM
  • I-A1. Deterministic tier order. Always (a) → (b) → (c) → (d). Never skip a tier, never re-order. Two reanchor runs over the same input must yield the same output.
  • I-A2. Orphan is recoverable, not terminal. A later regen can move an anchor back from orphaned to anchored / migratedFrom; the invariant from the Comment FSM that "orphan is a flag" applies here too.
  • I-A3. migratedFrom is only set when sectionId actually changes. Tier (b) does not set the field (section preserved); tier (c) does.
  • I-A4. Ties orphan, not guess. If tier (c) finds two sections above the Jaccard threshold, the anchor becomes orphaned rather than binding arbitrarily. Audit log records the tie.
  • I-A5. Telemetry is emitted exactly once per reanchor run. [comments] reanchor <scenario>/<doc> held=N migrated=M orphaned=K — a single summary line even if the log has 1000 comments.
  • I-A6. Deleted comments are invisible to reanchor. See I-C1: deleted is terminal; the anchor axis does not run on them.

5.4 Open Questions / Edge Cases

  • [ ] The Jaccard threshold (0.72) is drawn from §10 R-1. Needs a small corpus test to validate under realistic regen churn before locking the number.
  • [ ] If the same exact text appears twice in the new section (e.g. "the default") — tier (b) currently treats that as a hit at the first occurrence. Is that correct, or should duplicate hits orphan like tier-c ties? TODO: decide before MVP.
  • [ ] What about anchors whose exact was a block element (e.g. an entire <li>) that got split on regen? Tier (a) will miss; tier (b)/(c) will likely migrate; explicit test case needed.
  • [ ] Should section-ID collisions (the -2 / -3 collision resolver from §3) route through a separate ambiguous state, or fall into orphan? Current design treats them as distinct ids; no special state.

6. Session Lifecycle

Entity
Session (auth.js)

Every mutation route above checks a session first. On loopback, the session is implicit and bypassed (author="Host (local)"). For remote clients through the devtunnel, the flow is: anonymous → authenticated (HMAC-signed cookie, rolling TTL) → expired (TTL lapsed) or revoked (host-initiated). This FSM is owned by auth.js; the comment module consumes it via verifyCookie().

6.1 States & Diagram

Figure 6.1. Session lifecycle. Loopback skips the entire machine. Remote sessions roll their TTL on every valid request; plan_share_stop rotates the secret (design §10 R-9), invalidating every outstanding cookie in one shot.

6.3 Invariants

Invariants Session FSM
  • I-S1. Loopback never enters the FSM. The bypass is the first check; it cannot be disabled from a request. A remote actor who spoofs X-Forwarded-For cannot reach the bypass because the check is on remoteAddress.
  • I-S2. TTL is rolling, not sliding per-route. Every valid request refreshes the cookie exp. A user who polls every 5 minutes stays authenticated indefinitely until plan_share_stop or explicit logout.
  • I-S3. Revocation is atomic. plan_share_stop rotates the HMAC secret; there is no per-session revoke table. One action invalidates all cookies. Cheap and sound.
  • I-S4. Login failures are rate-limited. Without this, a devtunnel is a password-spray target. The RL key is IP + name; stderr audit shows every failure.
  • I-S5. Session name is stamped into comment author, not read from it. See I-C3.
  • I-S6. role: host is never client-settable. The server assigns the role at login time based on how the session was created (loopback, or an explicit host login gate). The intent:"revise" permission matrix depends on this.

6.4 Open Questions / Edge Cases

  • [ ] Concrete TTL length (30 min? 24h?). Design points at auth.js rolling TTL but the number is not in this scenario's design doc; needs to be pinned before test cases.
  • [ ] Explicit /logout route? Useful for test cases and for reviewers on shared machines. Not in design today.
  • [ ] Should role: host be assignable to a remote session (e.g. a second teammate who logs in with the host password)? Design treats host purely as loopback + reserved login name — confirm.
  • [ ] SSE cookie handoff: if the cookie rolls during an open SSE connection, does the next open after reconnect pick up the new cookie or fall back to anonymous? See §7.

7. SSE Client Lifecycle

Entity
SSE client (browser EventSource + server registry)

Each open doc holds one EventSource to GET /api/comments/:s/:d/stream. The server keeps a per-doc Set<Response> and flushes comment events to every member on each mutation, plus a ping every 30 seconds to keep the devtunnel warm. The browser auto-reconnects on error (built-in EventSource behaviour with exponential backoff up to ~10 s).

7.1 States & Diagram

Figure 7.1. SSE client lifecycle. disconnected is intentionally not terminal — the browser's EventSource retries with exponential backoff and re-enters opening until the session is revoked or the tab closes, at which point it terminates.

7.3 Invariants

Invariants SSE FSM
  • I-E1. At-most-once per event, per client. The server broadcasts on successful append. There is no replay on reconnect; clients re-fetch via GET /api/comments/:s/:d to reconcile. Users never see a duplicate "new comment" toast for the same JSONL line.
  • I-E2. Disconnected is not terminal. The browser retries with exponential backoff (up to ~10 s). UI reflects this with a non-modal banner; creation remains possible (drafts queue locally, see UC-12).
  • I-E3. Pings are opaque. event: ping carries data: {}; the widget ignores it. It exists only to reset devtunnel idle timeouts (design §4 tech choices).
  • I-E4. Revocation fans out in bounded time. When the HMAC secret rotates, every open stream receives a revoked frame in the next broadcast cycle; latency ≤ 30 s (the ping interval is the worst case).
  • I-E5. Auth gate is in-band. The SSE GET carries the same cookie as everything else; there is no query-string token, no Authorization header. See Design §9.
  • I-E6. Registry never leaks. Server res.on('close', ...) always removes the entry. Unit test: 100 open/close cycles leaves the Set empty.

7.4 Open Questions / Edge Cases

  • [ ] Should the widget synthesise a client-side replay (re-fetch full list) on reconnect? MVP: yes, quietly — but invariant I-E1 only holds if the re-fetch and the incoming live stream don't double-count during the transition. Needs a test.
  • [ ] How are revise status changes carried on SSE — as comment events (current design) or their own channel? If the former, the client needs to know to look at reviseStatus. Clarify in test cases.
  • [ ] Devtunnel flakiness during the 30 s ping window: if two pings are missed, the browser may not yet treat it as disconnected. Consider a watchdog that forces reconnect after 45 s of silence.
  • [ ] Back-pressure: what if a viewer has 10 docs open and 10 SSE streams? Per-session cap would be sensible; none in MVP.

8. Rate-Limit Bucket Lifecycle

Entity
Token bucket (per-session POST limiter)

Design §5 pins the bucket at 5 POST / session / 10 s (also mentioned as 10/min + 100/hr in §10 R-2 — implementation likely composes both). Conceptually, each bucket has tokens (0..capacity) and a lastRefill timestamp. Four states fold the two values: full (tokens == capacity), draining (0 < tokens < capacity), empty (tokens == 0, the next POST is 429), and refilling (transient — the refill math is running on a cold bucket that hasn't been touched in a long time).

8.1 States & Diagram

Figure 8.1. Token bucket lifecycle. refilling is a transient folding step: the server lazily recomputes tokens = min(cap, tokens + elapsed*rate) on each touch, then routes to full or draining based on the outcome.

8.3 Invariants

Invariants Bucket FSM
  • I-B1. Tokens never go negative. The draining → empty transition fires when the last token is drawn; subsequent POSTs never decrement.
  • I-B2. Tokens never exceed capacity. The refill math always clamps: min(cap, tokens + elapsed*rate).
  • I-B3. Refill is lazy, not timer-driven. There is no setInterval; the bucket is only touched when a request arrives. This keeps idle sessions free.
  • I-B4. Bucket loss on restart is benign. A restarted server grants a fresh bucket — worst case, an attacker gets capacity extra requests once. Acceptable because the attacker was also disconnected and must re-login.
  • I-B5. 429 never touches the log. The audit log is reserved for actual mutations and security-class denies. A 429 is a benign retry signal, not a security event.
  • I-B6. Loopback bypasses the bucket. The host cannot rate-limit itself; otherwise scripted workflows (/plan-revise batching 30 pending revises in a row) would trip the limiter pointlessly.

8.4 Open Questions / Edge Cases

  • [ ] Design §5 says 5/10s; §10 R-2 says 10/min + 100/hr. MVP probably composes both (short-burst and long-term). TODO: pin the composite formula before test cases.
  • [ ] Retry-After precision (whole seconds vs ms)? Browsers handle seconds fine; widget should floor.
  • [ ] Do PATCH / DELETE share the same bucket, or each has its own? Design reads "5 POST per session per 10s" — POST only. Other methods are currently unlimited; this may be deliberate (edits are rarer) or a gap.
  • [ ] SSE stream GET obviously should not consume bucket tokens. Confirm in implementation.

9. Cross-Entity Invariants

Some invariants constrain how two FSMs must line up. These are the ones most likely to produce compound bugs if left unchecked; each has a corresponding test class that the tester can expand into concrete cases.

Cross-invariants
  • X-1. Deleted parent ⇒ revise axis frozen. If a Comment moves to deleted, any revise state (pending/dispatched/proposed) is frozen. Accept/reject routes return 404.
  • X-2. Revoked session ⇒ SSE closed. The session transition to revoked fans out to every SSE in the session's doc map; each enters closed within one ping interval.
  • X-3. Empty bucket never produces a JSONL line. The bucket check is the first gate after auth; a 429 response precludes any Comment or Revise transition.
  • X-4. Anchor orphaned=true ⇒ Revise accept blocked. Even if the revise FSM is in proposed, the accept guard depends on a current anchor match; an orphaned anchor trips the drift guard and refuses to patch.
  • X-5. Loopback bypass is composable. Loopback bypasses both the Session FSM and the Bucket FSM; every other FSM still applies (Comment, Revise, Anchor, SSE). This is the single "god mode" for the host — nothing else elevates privilege.
  • X-6. Event order is preserved on disk but not guaranteed on the wire. JSONL is strictly append-ordered; SSE broadcasts are best-effort per-client. Two viewers may see mutations in slightly different orders; reconciliation is a GET-list pull after reconnect (see I-E1).

Tester hand-off: Every "never" arrow in the diagrams above corresponds to a negative test case. Every "(guard fails)" row in a transition table corresponds to an HTTP error response test. Every invariant (I-C1 … I-B6 and X-1 … X-6) is a property-style assertion the test harness should keep green across runs. When the Test Cases doc lands, link each assertion back here by id.
